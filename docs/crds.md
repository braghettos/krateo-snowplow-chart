# snowplow CRDs (chart repo)

The CRDs snowplow owns ship from the **`crds-subchart/`** chart (`krateo-snowplow-crd`,
`crds-subchart/Chart.yaml`), deployed as its own Composition independently of the app (see
[overview.md](overview.md)). This page is a **curated** field map â€” the authoritative, exhaustive
schema is the CRD YAML itself, and the Go source of truth is the code repo's `apis/templates/`.

## What this repo ships

This chart owns **one** CRD:

| Kind | Group/Version | Plural | File |
|------|---------------|--------|------|
| `RESTAction` | `templates.krateo.io/v1` | `restactions` (short `ra`) | `crds-subchart/templates/templates.krateo.io_restactions.yaml` |

> **There is no `Widget` CRD in this repo.** Widget is a frontend-widget type defined and rendered
> elsewhere (its Go types live in the code repo's `apis/templates/`, documented in
> [`howto/widgets.md`](https://github.com/braghettos/krateo-snowplow/blob/main/howto/widgets.md));
> the snowplow runtime resolves Widget CRs but the chart in *this* repo only publishes RESTAction.
> If you need the Widget schema, read the code repo, not here.

CRD subchart version: **`0.21.1`** (`crds-subchart/Chart.yaml:20`), pinned independently of the app
chart. The CRD carries `helm.sh/resource-policy: keep` so it survives chart uninstall
(`restactions.yaml:7`) and is generated by `controller-gen` `v0.17.3` (`restactions.yaml:6`).

## RESTAction â€” purpose

`RESTAction` lets you **declaratively define calls to APIs that may in turn depend on other calls**
(`restactions.yaml:31`). snowplow executes them via `/call`; responses must be JSON, and all HTTP
verbs are supported. Results are JQ-shaped and can be chained or iterated. It is the primitive the
portal uses to compose widget data from internal (apiserver) and external endpoints.

Shape: `spec.api[]` is an ordered list of stages, plus an optional global `spec.filter`
(`restactions.yaml:135-137`). `status` is open
(`x-kubernetes-preserve-unknown-fields: true`, `restactions.yaml:138-140`). `metadata` + `spec` are
required (`restactions.yaml:141-143`).

## RESTAction â€” key fields

### `spec.api[]` â€” the call stages (atomic list)

Each stage requires `name` (`restactions.yaml:123-125`). Key fields per stage:

| Field | Type | Purpose |
|-------|------|---------|
| `name` | string | Unique identifier; the stage's JSON result is stored under this key, read elsewhere as `.<name>`. |
| `path` | string | The request URI path (e.g. `/api/v1/namespaces`, or `/call?...` to invoke another RESTAction). |
| `verb` | string | HTTP method; defaults to GET when omitted. |
| `filter` | string | Optional per-stage JQ transform on this call's result. |
| `payload` | string | Optional request body. |
| `headers` | []string | Optional custom request headers (atomic list). |
| `endpointRef` | object `{name, namespace}` | Reference to an Endpoint (a Secret holding `server-url`/`token`) for external APIs; both fields required. |
| `dependsOn` | object `{name, iterator?}` | Sequence/iterate: `name` chains after a prior stage (linear Aâ†’Bâ†’C); `iterator` (a JQ path) runs the stage once per array element. |
| `exportJwt` | bool | Forward the caller's JWT to downstream calls. |
| `continueOnError` / `errorKey` | bool / string | RBAC-tolerant chaining: don't abort on a failing stage; stash the error under `errorKey`. |
| `userAccessFilter` | object | RBAC user-aware-filter mode for this stage (see below). |

(`restactions.yaml:66-122`)

### `spec.api[].userAccessFilter` â€” RBAC user-aware filtering

When present, the stage dispatches via the **snowplow ServiceAccount** (cluster-wide read) and the
returned result set is **re-filtered in-process through `EvaluateRBAC`** before being returned to
the caller (`restactions.yaml:36-51`). It is atomic â€” there is no per-mechanism toggle; absence
leaves pre-existing per-user-token dispatch unchanged.

| Field | Type | Purpose |
|-------|------|---------|
| `group` | string (required) | API group of the checked resource; `""` = core group (use `""` explicitly). |
| `verb` | string (required) | Lower-case RBAC verb checked per object (`get`/`list`/`watch`/â€¦). |
| `resource` | string | Static plural resource name (e.g. `namespaces`). Required **unless** `resourcesFrom` is set. |
| `resourcesFrom` | string | A JQ expression evaluated **once** against the full resolve dict, yielding a `[]string` of resource plurals â€” lets the checked set be runtime-discovered. OR-semantics across the set. |
| `namespaceFrom` | string | JQ path evaluated per returned object to derive the per-object namespace for the RBAC check. Defaults to `.metadata.namespace` (`restactions.yaml:58-59`). |

Validation rules (CEL `x-kubernetes-validations`):

- **`userAccessFilter` requires exactly one of `resource` / `resourcesFrom`** â€”
  `has(self.resource) != has(self.resourcesFrom)` (`restactions.yaml:117-119`). This is the XOR
  rule the `0.21.1` schema refresh added.
- **`userAccessFilter` is only allowed on read-verb stages** (GET/HEAD, case-insensitive) â€” CRUD
  verbs under filter scope would expose mutation (`restactions.yaml:127-128`).
- **A `userAccessFilter` stage must NOT set `exportJwt: true`** â€” would leak the raw JWT through the
  filtered response (`restactions.yaml:129-130`).
- A `userAccessFilter` must specify a non-empty `verb` and a non-empty
  `resource`/`resourcesFrom` â€” a degenerate filter would collapse the SubjectAccessReview to a
  wildcard (`restactions.yaml:131-132`).

### `spec.filter`

Optional global JQ filter applied to the final, fully-resolved result (`restactions.yaml:135-136`).
